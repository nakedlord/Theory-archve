# SynTax Patch v4.6 — Complexity Governance Rule

Статус: active architecture-governance patch

## Проблема

Архитектура может расти быстрее, чем растёт её различающая сила.

Каждый новый слой кажется полезным локально, но вместе они создают architecture bloat.

## Добавленная конструкция

Complexity Governance Rule.

## Определение

Любое увеличение архитектурной сложности допустимо только при явном приросте различающей силы относительно стоимости усложнения.

## Что оценивается

- новый слой;
- новая runtime-stage;
- новая метрика;
- новый error class;
- новый audit-механизм;
- новый обязательный пакет передачи.

## Главный вопрос

Нельзя ли получить тот же результат уточнением существующего слоя, а не добавлением нового?

## Критерии пользы

- новое различение;
- устранение повторяющегося сбоя;
- улучшение runtime metrics;
- рост auditability;
- снижение ambiguity.

## Критерии стоимости

- рост числа обязательных стадий;
- усложнение воспроизводимости;
- нагрузка на validation;
- риск дублирования функций.

## Зачем это нужно

Патч защищает SynTax от раздувания.

## Риск

Если правило применять слишком жёстко, оно может блокировать нужные архитектурные улучшения.

## Связи

- Patch Protocol;
- Audit Protocol;
- Exploration Budget;
- Operator Registry.
